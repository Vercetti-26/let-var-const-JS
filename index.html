If you've written JavaScript for more than a day, you've typed one of these three words: var, let, or const. They all declare variables. But they are wildly different creatures under the surface — and mixing them up is the silent cause of some of the most baffling bugs in JavaScript codebases.

This post is the definitive breakdown. We'll go through the history of why let and const were even created, how hoisting works, what block scope actually means, and the practical rules you should follow today.

"Choosing var in modern JS is like using a typewriter because you know how to type."

A Quick History — Why Were let & const Even Created?

JavaScript was written in 10 days in 1995 by Brendan Eich. Back then, var was the only way to declare a variable. It worked — sort of. But it had quirks that caused real problems at scale.

1995 — ES1
The Age of var
JavaScript ships with only var. It has function scope, gets hoisted to the top of its function, and can be re-declared freely. Nobody thinks this is a problem — codebases are tiny.
2000s — jQuery era
The Problems Surface
As SPAs and large codebases grow, developers hit var's scoping bugs constantly. Loop variables leak, hoisting creates undefined-before-assigned bugs, and re-declarations cause silent overwrites.
2015 — ES6 (ES2015)
let and const Arrive
TC39 — the committee that governs JavaScript — introduces let and const as part of ES6. Both are block-scoped, not hoisted in the same dangerous way, and solve nearly every pain point of var.
Today — 2026
var is Legacy
Modern JS style guides (Airbnb, Google, StandardJS) all flag var as legacy. ESLint's no-var rule is standard. const and let are the only keywords you should be writing.

The Three Keywords at a Glance

var
The OG · ES1
Function-scoped, hoisted, re-declarable, and mutable. The original — and the most dangerous for modern code.
let
The Flex · ES6
Block-scoped, not re-declarable in same scope, mutable. Use when a value needs to change over time.
const
The Rock · ES6
Block-scoped, immutable binding, must be initialized. Your default choice — use it until you have a reason not to.

Scope: The Core Difference

Scope defines where a variable is accessible. This is where var and let/const diverge most dramatically.

var is function-scoped

A var variable lives inside the entire function it was declared in — or globally if declared outside any function. If you declare it inside an if block or a for loop, it still leaks out to the whole function.

JavaScript — var scope leak
function exampleVar() {
  if (true) {
    var message = "I'm inside the if block";
  }
  // 😱 This still works — var leaks out of the if block
  console.log(message); // "I'm inside the if block"
}

let & const are block-scoped

A "block" is anything wrapped in curly braces: if, for, while, function bodies, or even a standalone { }. With let and const, variables stay inside that block — no leaking.

JavaScript — let block scope
function exampleLet() {
  if (true) {
    let message = "I'm block scoped";
    console.log(message); // ✅ Works fine here
  }
  // ❌ ReferenceError: message is not defined
  console.log(message);
}
💡
The Loop Bug — Classic var Trap The infamous var loop bug: when you use var i in a for loop with setTimeout, all callbacks share the same i — which is already at its final value by the time they run. Using let i creates a fresh i for every iteration. This single difference was one of the biggest motivators for let.

Hoisting: The Hidden Behavior

Hoisting is JavaScript's behavior of moving declarations to the top of their scope before code runs. All three keywords are technically hoisted — but what gets hoisted is very different.

var: hoisted AND initialized to undefined

When var is hoisted, JavaScript also initializes it to undefined. That means you can access a var variable before the line where you declared it — you just get undefined instead of an error. This is a frequent source of subtle bugs.

JavaScript — var hoisting
console.log(name); // undefined (no error!)
var name = "Arjun";
console.log(name); // "Arjun"

// JS internally treats it as:
var name = undefined; // hoisted here
console.log(name);
name = "Arjun";
console.log(name);

let & const: hoisted but NOT initialized (Temporal Dead Zone)

Both let and const are hoisted to the top of their block — but they are not initialized. The period between the start of the block and the actual declaration line is called the Temporal Dead Zone (TDZ). Accessing the variable in this zone throws a ReferenceError.

JavaScript — Temporal Dead Zone
// ─── Temporal Dead Zone for 'score' starts here ───
console.log(score); // ❌ ReferenceError: Cannot access 'score' before initialization
let score = 100;
// ─── TDZ ends here ───
console.log(score); // ✅ 100
⚠️
TDZ is a Feature, Not a Bug The Temporal Dead Zone was deliberately designed to catch errors. If you're accessing a variable before declaring it, that's almost certainly a bug. Getting a hard ReferenceError is far better than silently getting undefined and debugging for hours.

Re-declaration vs Reassignment

These two concepts trip people up. Re-declaration means creating a new variable with the same name. Reassignment means changing the value of an existing variable.

JavaScript — Re-declaration vs Reassignment
// ── var: both allowed ──────────────────────────────
var city = "Mumbai";
var city = "Delhi";  // ✅ re-declaration — no error (dangerous!)
city = "Bangalore";  // ✅ reassignment

// ── let: only reassignment allowed ────────────────
let count = 1;
let count = 2;  // ❌ SyntaxError: Identifier 'count' has already been declared
count = 2;      // ✅ reassignment is fine

// ── const: neither allowed ────────────────────────
const PI = 3.14;
const PI = 3;   // ❌ SyntaxError
PI = 3;         // ❌ TypeError: Assignment to constant variable

const with Objects & Arrays — A Common Misconception

const makes the binding immutable — meaning you can't point the variable at a new value. But if that value is an object or array, the contents can still be mutated. This catches many developers off guard.

JavaScript — const with objects
const user = { name: "Riya", age: 25 };

user.age = 26;           // ✅ Mutating properties is fine
user.city = "Chennai";  // ✅ Adding new properties is fine

user = { name: "Priya" }; // ❌ TypeError — can't reassign the binding

// Same for arrays:
const nums = [1, 2, 3];
nums.push(4);  // ✅ Mutating the array is fine
nums = [5];   // ❌ TypeError
🔒
Want True Immutability? If you want to completely freeze an object so its properties can't change either, use Object.freeze(user). Note that this is shallow — nested objects inside the frozen object are still mutable.

The Rules

🏆 The Modern Variable Decision Framework

  • 1 Use const by default for everything. If you know a value won't need to change — use const. This is your starting point 90% of the time.
  • 2 Switch to let only when you need to reassign. Counters, accumulators, loop variables that get re-assigned, toggling flags — these are let territory.
  • 3 Never use var in new code. There is no modern use case where var is preferable to let or const. It exists for backward compatibility only.
  • 4 Remember: const doesn't mean immutable data — it means immutable binding. Objects and arrays declared with const can still have their contents changed.

Real-World Code — Seeing it in Practice

JavaScript — Modern best practices
// ✅ API URL never changes — const
const API_URL = "https://api.example.com/v1";

// ✅ Object reference stays the same — const
const config = {
  timeout: 5000,
  retries: 3,
};

// ✅ Counter changes — let
let requestCount = 0;
requestCount++;

// ✅ Loop variable — let (new binding per iteration)
const items = ["a", "b", "c"];
for (let i = 0; i < items.length; i++) {
  console.log(items[i]);
}

// ✅ Arrow functions with const
const fetchUser = async (id) => {
  const res = await fetch(`${API_URL}/users/${id}`);
  const data = await res.json();
  return data;
};
Takeaway

var is JavaScript's past. It served its purpose in a simpler era, but its function-scope behavior and permissive hoisting are traps in any non-trivial codebase. The introduction of let and const in ES6 wasn't just syntax sugar — it was a fundamental fix to how JavaScript developers reason about state and scope.

The mental model is simple: start with const, reach for let when mutation is required, and leave var to the history books. Your future self — and every developer who reads your code — will thank you.
